# Cell IR: 基于传播器模型的显式数据流中间表示

## 动机：绕过 SSA 的迂回

传统编译器将结构化的表达式语言（如 Lisp）先转换为带有 `goto` 和基本块的控制流图（CFG），再费力构建静态单赋值形式（SSA）进行分析和优化。这一“先破坏结构，再恢复信息”的过程源于历史上对控制流的统一处理，但对于没有显式跳转、以嵌套表达式为主的语言，我们完全可以采用更直接的路线。

**核心质疑**：为什么要把不带 `goto` 的代码转换成带 `goto` 的 CFG 再去分析？

**解答**：让中间表示（IR）本身就是一个显式命名的 **propagator network**——每个指令定义一个 **cell**，操作数通过 cell 名称间接引用，控制依赖降级为数据选择，信息在整个网络中增量传播。

## IR 语法

Cell IR 由一系列 cell 定义组成。每个定义引入一个全局唯一的 cell 名（以 `%` 开头），并包含一条指令。

### 指令格式

    (cell %name <instr>)

其中 `<instr>` 可以是以下形式之一：

- **带局部别名的绑定**：`(= <local-name> <op-expr>)`
  例如：`(cell %x.1 (= x (iadd 1 2)))`
  这里 `%x.1` 是全局 cell 名，`x` 是仅供人类阅读的局部别名，在后续引用时仍需使用 `%x.1`。
  *别名部分可省略，直接写成 `(cell %name <op-expr>)`。*

- **直接指令**：`(<op> <arg>*)`
  例如：`(cell %t2 (iadd %t1 %b))`

#### 操作符列表

| 操作符 | 含义 | 示例 |
|--------|------|------|
| `const` | 常量 | `(cell %c (const 42))` |
| 算术/逻辑原语 | `iadd`, `isub`, `imul`, `ieq`, `ilt` 等 | `(cell %s (iadd %a %b))` |
| `call` | 函数调用 | `(cell %r (call %f %a %b))` |
| `if` | 条件选择（数据级 φ） | `(cell %w (if %cond %then %else))` |
| `lambda` | 函数抽象 | `(cell %lam (lambda (%x) %body))` |
| `param` | 参数引用 | `(cell %p (param 0))` |
| `propagator` | 显式自定义传播器 | `(cell %t (propagator (%in) (%out) <handler>))` |

#### 操作数约定
- 操作数是字面量（整数、布尔、字符串等）或其它 cell 的全局名（如 `%t1`）。
- 函数参数在 `lambda` 内部通过 `(param <index>)` 获得对应的参数 cell，也可用 `%param0` 等预定义名称直接引用。

## 翻译示例

### 简单算术
源表达式：`(+ (* a 2) b)`

    (cell %t1 (imul %a 2))
    (cell %t2 (iadd %t1 %b))

如果使用局部别名：

    (cell %t1 (= t1 (imul %a 2)))
    (cell %t2 (= t2 (iadd %t1 %b)))

### 条件表达式
源表达式：`(if flag (+ x 1) (- x 1))`

    (cell %then (iadd %x 1))
    (cell %else (isub %x 1))
    (cell %result (if %flag %then %else))

`if` 指令是一个 **数据选择器**：根据 `%flag` 的值选择 `%then` 或 `%else`，完全消除了对 φ 节点和控制流边的需求。

### 函数与闭包
源表达式：`(lambda (x) (+ x a))`

    (cell %lam (lambda (%x) %body))
    (cell %x (param 0))          ; 对应 lambda 的第一个参数
    (cell %body (iadd %x %a))    ; 自由变量 %a 直接引用外层 cell

`%lam` cell 的值是一个闭包，包含参数列表、体 cell 引用以及捕获的自由变量 cell 列表。

## 核心特性

### 1. 显式 def-use 图
IR 本身就是一张有向图：节点是 cell，边是操作数中的 cell 引用。无需额外的 use-def 分析，依赖关系一目了然。

### 2. 无控制流指令
不存在跳转或基本块。唯一的值合并点就是 `if` cell，它是一个纯数据操作。程序的执行顺序由数据依赖拓扑排序决定，与控制流分离。

### 3. 原生支持“未知”与增量传播
每个 cell 除了存储值，还可携带类型、常量状态、值区间等分析信息，初始可为“未知”。当某个 cell 的信息更新时，仅触发依赖它的传播器重新计算。这使得编译器可以按需、增量地进行分析，非常适合 IDE 和渐进式编译场景。

### 4. 多向传播
传播器可同时注册正向和反向行为。例如类型推导：
- 正向：当操作数类型已知，推导结果类型。
- 反向：当结果类型被外界要求为 `int` 时，反向对操作数施加 `int` 约束。
这统一了正向数据流分析和反向约束求解。

### 5. 可编程传播器
`propagator` 指令允许在 IR 中直接嵌入分析逻辑。常量折叠、死代码消除、类型推断等优化都可以写成传播器，与程序自身的计算指令共存于同一网络。编译器优化不再是一个外部遍历器，而是一组与程序 IR 无缝协作的自治代理。

## 与 SSA 路线的对比

| 特性 | 传统 SSA / CFG | Cell IR |
|------|----------------|---------|
| 基本结构 | 控制流图 + 基本块 | 平坦的 cell 列表，无块结构 |
| 值合并 | φ 函数（需支配边界计算） | `if` 数据选择器 |
| 控制依赖 | 显式跳转、分支指令 | 通过 `if` 隐式表达 |
| def-use 信息 | 需额外分析重建 | 直接编码在操作数中 |
| 增量分析 | 通常需要重新遍历整个函数 | 天生支持增量传播 |
| 分析与优化 | 外部 pass，遍历 IR | 嵌入网络的传播器，与 IR 协同 |

## 对静态分析的意义

Cell IR 将静态分析从“**在控制流图上迭代数据流方程**”转变为“**在显式依赖网络上管理多个交互传播器的增量收敛过程**”。分析信息（类型、值区间、别名等）直接存放在 cell 中，不再需要额外的映射表。这极大地简化了编译器架构，同时天然支持按需、增量、多向分析，为类 Lisp 语言的交互式开发和元编程提供了统一的底层设施。

## 结语

Cell IR 是一种去除了控制流迷雾、回归表达式本质的中间表示。它把程序表示为一个命名的计算依赖图，既是可执行的规约，也是分析与优化的活性网络。通过将每个指令封装为一个 cell，并采用传播器模型进行信息流动，我们得以绕过传统 SSA 构造的复杂性，获得一种更简洁、更增量的编译器中间层设计。
